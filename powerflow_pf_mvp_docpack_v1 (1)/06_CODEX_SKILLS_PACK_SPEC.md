# 06 — Codex Skill Pack Spec (pf, MVP)

## 0) Зачем skill pack

Skill pack нужен, чтобы:
- в Codex была **единая точка входа** (`$pf`)
- работало автодополнение/выбор навыков
- ассистент действовал по одинаковому процессу (без “импровизаций”)
- запрещалось писать ad-hoc оркестрационные скрипты

Codex skills используют progressive disclosure: Codex видит `name/description` каждого навыка и читает полный `SKILL.md` только при выборе навыка. Поэтому можно держать много навыков без раздувания контекста.

---

## 1) Требования Codex к skills (контракт)

- Skill = директория внутри `.agents/skills/**`
- Внутри обязателен файл `SKILL.md`
- `SKILL.md` начинается с YAML front matter:
  - `name: ...`
  - `description: ...`

---

## 2) Набор skills (MVP)

### 2.1. `$pf` (root entrypoint)
Роль: “router / менеджер”.
Задача: всегда начинать с `pf status --json`, затем предложить пользователю действия менеджерским языком и вести по flow.

### 2.2. `$pf-intake`
Роль: intake / routing.
Задача: принять большую задачу (спеку), задать вопросы, определить:
- single-module patch или mission multi-module
- какие модули затрагиваем (1–3 на milestone-1)
- что считать “готово” (acceptance criteria)
Далее создать `mission` + `task` записи через pf CLI.

### 2.3. `$pf-planner`
Роль: планирование.
Задача: в рамках одного модуля создать:
- варианты A/B/C (простыми словами)
- рекомендованный вариант и почему
- scope / non-scope
- “как проверяем” (команды verify)
- разбиение на 1–5 шагов (slices)
Записать PLAN.md и обновить pf state.

### 2.4. `$pf-plan-review`
Роль: semantic review плана.
Задача: критически проверить план:
- ясность (для менеджера)
- риски/альтернативы
- нет ли “всё и сразу”
- есть ли check/verify
Результат: либо замечания, либо “plan reviewed ok”.

### 2.5. `$pf-executor`
Роль: выполнение.
Задача: выполнять только в рамках module scope.
Правила:
- всегда начать с `pf context build --intent execute`
- если нужно трогать другой модуль — остановиться и спросить пользователя, предложив создать task в другом модуле
- после изменений: запустить verify команды (из runbook/плана), зафиксировать результаты событиями
- затем `/review`

### 2.6. `$pf-retro`
Роль: ретро и улучшения.
Задача: после сдачи:
- собрать проблемы (incidents)
- оформить retro
- предложить улучшения в `.pf/local/` и/или PKM

## 3) AGENTS.md (контракт поведения ассистента)

AGENTS.md должен:
- задавать стиль общения: сухо, понятно, без “поощрений”
- запрещать “скриптовую оркестрацию”
- требовать **всегда** начинать с `pf status --json` и работать через `pf context build`
- требовать спрашивать пользователя, если что-то неясно (без галлюцинаций)
- требовать держать module boundaries

Codex применяет AGENTS.md по иерархии: root → … → CWD (ближайшие инструкции имеют приоритет).
Для модулей можно создавать `AGENTS.override.md` внутри `.pf/modules/<module>/` и/или рядом с root_path модуля (если это удобно).

---

## 4) Использование встроенных возможностей Codex (обязательное)

### 4.1. Permissions / approval modes
Рекомендованный режим:
- до утверждения плана: **Read-only**
- после утверждения: **Auto**
- Full Access — только если реально нужен.

В Codex CLI это делается slash-командой `/permissions`.

### 4.2. Code review
Перед “готово” ассистент обязан запускать `/review` (встроенный ревьюер Codex) и учитывать замечания.

---

## 5) Артефакты, которые skills обязаны создавать

- `.pf/modules/<module>/PLAN.md` — план
- `.pf/modules/<module>/DECISIONS.md` — важные решения (коротко)
- `.pf/modules/<module>/KNOWLEDGE.md` — runbook команды (обновляется по результатам)
- `.pf/modules/<module>/RETRO/<date>.md` — ретро
- `.pf/artifacts/bundles/*.json/.md` — контекст-бандлы (pf генерирует)
- `.pf/artifacts/logs/*` — логи verify команд (pf регистрирует)

---

## 6) Анти-галлюцинации (правила)

Skill pack должен навязывать:
- если данных недостаточно → задавать вопросы пользователю
- не придумывать команды тестов, если не найдено подтверждение (runbook/план/README) → спросить или предложить варианты
- в конце каждого шага писать, что сделано и что дальше (NEXT)
