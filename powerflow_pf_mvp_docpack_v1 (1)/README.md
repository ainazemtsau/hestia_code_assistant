# PowerFlow (pf) — MVP Docpack (Codex-first)

Дата: 2026-03-01  
Цель: **с нуля** реализовать PowerFlow (pf) как *data kernel + context packaging* для AI code assistants (начинаем с **Codex**), без самописного «оркестратора».

Этот архив — **единственный источник правды** для реализации MVP. Предполагается, что Codex/ассистенты знают **только то, что внутри этого архива**.

---

## 0) Ключевой принцип (не обсуждается)

pf **не** пытается:
- угадывать архитектуру проекта,
- выполнять “wizard” логикой кода,
- писать сложные проверки и “супер-движок”.

pf **делает только**:
1) хранит **структурированные данные** (SQLite) о проекте/миссиях/задачах/модулях/решениях/инцидентах;
2) хранит и индексирует **память** (PKM, docs freshness) по модулям;
3) собирает **детерминированный Context Bundle** (строго ограниченный) под текущий шаг;
4) печатает **понятный статус** и **NEXT command**.

Все рассуждения, планирование, выбор вариантов, ревью “по смыслу” — это задача **host ассистента** (Codex), реализованная через **skills + AGENTS.md**.

---

## 1) Что считается «готовым MVP»

MVP готов, если выполняется Acceptance A:

- новый репозиторий → `./pf init`
- Codex видит skills и AGENTS.md
- пользователь вводит большую задачу (как менеджер)
- ассистент:
  - создаёт mission
  - выделяет 1–3 модуля
  - делает план по модулю (варианты A/B/C)
  - получает подтверждение пользователя
  - выполняет изменения (в рамках модуля)
  - запускает тесты/проверки (команды берёт из runbook/PKM либо спрашивает)
  - делает `/review`
  - выдаёт отчёт + просит пользователя проверить
  - делает retro и обновляет память (PKM/docs)
- при этом ассистент **не** пишет “временные python-скрипты для оркестрации”, а использует только pf CLI + стандартные инструменты Codex.

---

## 2) Как работать по плану

Папка `phases/` содержит пофазный план (каждая фаза — отдельный, проверяемый кусок).

**Правило:** не начинать следующую фазу, пока acceptance текущей не пройдены.

Рекомендуемый режим:
- 1 фаза = 1 PR
- в PR обязательно:
  - обновлённые тесты
  - CLI транскрипт (минимум 1) в `examples/cli_transcripts/`
  - чеклист acceptance из файла фазы (заполненный)

## 3) Структура архива

- `00_SCOPE_AND_PRINCIPLES.md` — ограничения и принципы (в т.ч. NO-SCRIPTS).
- `01_ARCHITECTURE_OVERVIEW.md` — архитектура pf (data kernel).
- `02_DATA_MODEL_SQLITE.md` — таблицы/индексы/DDL.
- `03_CLI_SPEC.md` — команды, форматы вывода, коды ошибок.
- `04_MEMORY_CONTEXT_SPEC.md` — детальная механика памяти/контекста (без «магии»).
- `05_WORKFLOW_EXAMPLES.md` — живые сценарии “как это работает”.
- `06_CODEX_SKILLS_PACK_SPEC.md` — требования к skills/AGENTS.md (Codex-first).
- `07_SECURITY_GUARDRAILS.md` — guardrails: модульные границы, “не сканировать всё”.
- `08_ACCEPTANCE_AND_EVAL.md` — Acceptance A/B + регрессии.

- `templates/` — шаблоны документов и SKILL.md.
- `examples/` — примеры ожидаемых выводов.

---

## 4) Технологические решения MVP

- Язык: **Python 3.11+**, **stdlib-only** (без внешних зависимостей).
- Хранилище: **SQLite** в `.pf/state.db`.
- Портируемость: macOS/Linux (Windows можно позже).
- Host: **Codex** (другие ассистенты — через MCP в будущем).

---

## 5) Важные запреты (NO-SCRIPTS)

- ❌ Нельзя добавлять “оркестрационные” скрипты, которые пытаются заменить навыки/assistance (визарды, роутинг, планирование).
- ✅ Можно добавлять код/скрипты *только* для:
  - чтения/записи SQLite,
  - детерминированного сборщика контекста,
  - детектора stale docs,
  - печати status/next.

Если ассистенту “не хватает команды” — он должен:
- записать incident,
- предложить минимальное расширение pf CLI или skill,
- но не писать ad-hoc tooling.
