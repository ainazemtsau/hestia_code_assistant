# 00 — Scope и принципы (MVP)

## 0) Контекст

Мы строим **pf** как слой данных и упаковки контекста для AI code assistants (начинаем с Codex).
pf **не заменяет** ассистента и **не реализует** “разум” в коде.

---

## 1) Что мы делаем (MVP)

pf делает только 4 вещи:

1) **State / SSOT**: пишет события и хранит состояние проекта в SQLite (append-only events + компактные таблицы состояния).
2) **Module-first memory**: хранит знания/документы/решения по модулям и помечает их актуальность (stale).
3) **Context Builder**: детерминированно собирает ограниченный Context Bundle под intent (plan/execute/review/retro/status).
4) **Status + NEXT**: печатает понятный статус (для менеджера) и один следующий шаг.

---

## 2) Чего НЕ делаем (жёстко)

- ❌ Не делаем “оркестратор” планирования/роутинга в коде.
- ❌ Не делаем авто-архитектурный анализ больших репозиториев “до конца”.
- ❌ Не делаем 1000 собственных проверок/гейтов.
- ❌ Не делаем отдельную систему ревью — используем возможности host (например `/review` в Codex).
- ❌ Не плодим ad-hoc скрипты (Python/Node/…).

---

## 3) NO-SCRIPTS правило (важное)

### 3.1. Разрешённый код
Разрешён только код, который:
- читает/пишет SQLite;
- строит контекст из уже известных источников по правилам (deterministic);
- считает fingerprint и stale для docs;
- печатает status/next;
- валидирует целостность данных.

### 3.2. Запрещённый код
Запрещён код, который:
- пытается “вести диалог” вместо ассистента;
- “угадывает” модули/архитектуру/план;
- принимает продуктовые решения;
- делает “многошаговый wizard” логикой программы.

---

## 4) UX инварианты (то, что всегда должно быть)

1) **Одна точка входа**: пользователь почти всегда использует `./pf` (и/или `$pf` skill).
2) **NEXT**: каждый `pf status` выводит **одну** рекомендуемую команду next (и 1–2 альтернативы опционально).
3) **Модульность**: работа идёт в scope одного модуля; переход в другой модуль — явный.
4) **Не сканировать всё**: контекст собирается строго через `pf context build`, с budget и ограничением по путям.
5) **Актуальность docs**: любые “важные документы” должны иметь источники и stale-маркировку.

---

## 5) Расширяемость (после MVP)

- MCP сервер для Claude Code/Gemini CLI: pf становится источником tools/resources/prompts (без изменения ядра данных).
- Дополнительные виды памяти (векторы/графы) — только как плагины, не в MVP.
