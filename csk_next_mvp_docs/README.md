# CSK‑Next — MVP DocPack (для реализации ядра)

Версия: **MVP-docpack v0.1**  
Дата: **2026-02-21**  
Язык: **RU (с англ. идентификаторами в API/CLI)**

## Что это
Этот архив — **полный набор спецификаций** для реализации MVP ядра workflow **CSK‑Next** + минимального ядра **Event Log / Context Builder / PKM (runbook v0)**.

Команда разработки (включая code assistants) должна исходить из правила:
- **все, что нужно для реализации — находится здесь**;
- **если чего-то нет здесь — не “догадываться”, а добавлять как ADR/обсуждение**.

## MVP‑цели (что должно заработать end‑to‑end)
1) **Single entrypoint**: `csk` в root показывает статус и следующий шаг.
2) **Module‑first / Worktree‑first**: multi‑module задачи разворачиваются в worktree по модулю.
3) **Planning с человеком**: план и слайсы фиксируются артефактами, дальше исполнение автономно.
4) **Жёсткие гейты**: нельзя случайно “проскочить” план/ready.
5) **SSOT = Event Log**: каждое важное действие/наблюдение пишется событием.
6) **Context Bundle**: детерминированная сборка контекста с provenance (v1 — без тяжёлых индексов).
7) **PKM v0**: минимальный runbook (как запускать verify) выводится из событий.

## Негативные цели (что НЕ делаем в MVP)
- Нет vector DB / graph DB / Tree‑sitter / LSP / LSIF.
- Нет распределённой мульти‑агентности и CRDT.
- Нет “авто‑PR/merge”.
- Нет сложной платформы обновлений engine; только соблюдаем слойность (engine/state/overlay).

## Быстрый навигатор
- `01_PRODUCT_NORTH_STAR.md` — UX/продуктовые инварианты
- `02_ARCHITECTURE.md` — 3 слоя, SSOT, state machine
- `06_CLI_SPEC.md` — команда `csk`, подкоманды, формат вывода
- `08_GATES_SPEC.md` — Plan Gate / Slice Gate / Ready Gate
- `12_PHASE_PLAN/` — план разработки по фазам (атомарные куски)
- `templates/` — шаблоны артефактов и skills
- `examples/` — примеры миссии, статуса, outputs

## Как вести разработку
1) Идти по фазам из `12_PHASE_PLAN/` строго по порядку.
2) Каждая фаза:
   - добавляет **маленький работающий инкремент**,
   - заканчивается **checklist валидации**,
   - фиксирует решения (если нужно) в `ADR/`.

## Термины (минимум)
- **Mission** — большой запрос (может затрагивать много модулей).
- **Milestone** — итерация миссии; в MVP активен только milestone‑1.
- **Module** — основная единица контекста и worktree.
- **Task** — работа в рамках модуля.
- **Slice** — атомарный шаг исполнения внутри Task.
- **Gate** — проверка, которая блокирует переходы без доказательств.
- **Proof Pack** — артефакты/логи, подтверждающие прохождение гейтов.
- **Event Log** — append‑only журнал событий (SSOT).
